01

原理分析

Fusion 的收益不来自"更大的模型",而来自把多条相互独立的推理路径,经过一个会做结构化判断的合成步骤,收敛成一条答案。下面拆开它为什么有效、收益主要来自哪里,以及随之而来的几条不可妥协的设计约束。

1.1 误差去相关:为什么"一组模型"能赢"一个模型"

单个前沿模型的错误是系统性的——它在某类问题上稳定地漏掉同一个角度。把同一个提问交给架构、训练数据、工具调用策略各不相同的几个模型,它们的错误不再高度相关。OpenRouter 在 DRACO 深度研究基准上的结果印证了这一点:Fable 5 + GPT-5.5 融合得 69.0%,超过任何单模型(Fable 5 单独 65.3%);三个便宜模型(Gemini 3 Flash + Kimi K2.6 + DeepSeek V4 Pro)的预算面板以约一半成本逼近前沿单模型。这与"多元团队在复杂问题上胜过单个专家"是同一个机理。

1.2 合成步骤本身才是杠杆

一个反直觉但关键的发现:把 Opus 4.8 和它自己组成两模型面板、仍由 Opus 4.8 合成,得分 65.5%,比单独的 Opus 4.8(58.8%)高 6.7 分。同一个模型跑两遍会产生不同的推理路径、不同的工具调用、不同的取材——这说明相当一部分提升来自合成这一步,而不仅是模型多样性。

设计含义 合成不是把几段文本拼起来。裁判先产出一份结构化分析(共识 / 矛盾 / 部分覆盖 / 独特见解 / 盲点), 合成步骤据此重写。本系统把这份中间表示当作一等公民产物(可审计的 JSON),而非顺手丢弃的过渡态。

1.3 由 caller(调用方)写最终答案

文章里一个容易被忽略的架构点:裁判负责蒸馏,但写最终答案的是调用方模型(caller)。二者默认是同一个模型,但当你显式分离时,这个区别决定了"谁的笔触落在最终答案上"。本实现中 orchestrator 把 config.caller(默认等于 judge)传给合成步骤,正是这条原则的落地。

1.4 合成阶段冻结证据(关闭 web 工具)

面板阶段每个模型都可以联网检索;但到了合成阶段,所有 web 工具被刻意关闭。理由是可审计性:最终答案只能基于面板已经收集、裁判已经记录的证据,不能在最后一刻引入未经面板与裁判审视的新材料。这让"答案依据什么"可被完整回放。这是正确性约束,不是风格偏好——它在代码里由 toolset_for_phase(SYNTHESIS) == () 强制保证。

1.5 选择性触发:贵,所以要值得

Fusion 比单次调用慢约 2–3 倍,且按各次补全累加计费(每个面板 + 裁判 + 合成)。因此它不是默认路径,而是模型在"判断这个问题值得多花时间和钱"时才调用的工具——架构决策、最佳实践研究、多领域评审。把这种选择性当作设计本身,而不是限制。

适合

  • 深度研究、多领域批判、选型/架构决策
  • 答错代价高、或用户明确要"多模型共识/第二意见"

不适合

  • 短战术指令、对延迟敏感的对话
  • 答案明确的转换/摘要——单模型一次更快更省

03

评测系统

这是 skill 的"评测标准"。它完整复用 OpenRouter 在文章中采用的 DRACO 方法学,并在其上扩到全部 10 个领域,再加两类专项:agent 长程(pass^k 可靠性)与半导体/DRAM。目标是为你自己的任务和模型回答一个问题:Fusion 到底赢没赢单模型、赢多少、花多少代价、值不值。

3.1 加权评分:四类标准 + 负权重 + 0–100 归一化

每个任务带一份评分标准,跨四个类别加权:Factual Accuracy(约 20 条)、Breadth & Depth(约 9 条)、Presentation Quality(约 6 条)、Citation Quality(约 5 条),共约 39 条。关键在于标准可带负权重:命中一条负权重标准意味着回答犯了那个错(如危险医疗建议),要扣分。这正是"无法靠啰嗦、自信地说错来刷分"的机制。

方法学要点(忠实复刻)

  • 每条标准独立判 3 次,取均值。裁判模型逐条判"满足/不满足",harness 跑 N=3 遍平均,落在 _grade_avg。
  • 相对差才是可信信号。绝对分会随裁判模型变动 10–25 分,但"Fusion vs 单模型"的相对排名稳定——所以 verdict 以相对 lift 为准,而非绝对分。
  • 污染防护。每个任务的 excluded_domains 透传给面板 web 工具,正如文章排除了托管评分标准的域名,防止模型在线"抄到"标准。
  • 公平对比。Fusion 与单模型基线用完全相同的工具集(复用 panel._run_one);唯一差别是是否做多模型合成。

3.2 套件一:DRACO 十领域深度研究

扩到文章列出的全部 10 个领域,以可机检的探测器(any/all/none 正则)编码事实/覆盖/引用/惩罚标准,使同一套标准既能驱动离线 RuleGrader,也能交给真实的 LLM 裁判。

3.3 套件二:Agent 长程专项(pass^k 可靠性)

文章自己点明:DRACO 不含长程任务,而长程恰是 Fable 的强项。所以长程要单独测,且换一套指标。参考最先进的 agent 基准 τ-bench(Yao 等, 2024):它提出 pass^k = 在重复的全部 k 次尝试中都成功的概率,并暴露出惊人的可靠性鸿沟——GPT-4o 平均成功率 >60%(pass@1),但零售域 pass^8 < 25%。pass^k = pᵏ 指数衰减,对"最坏情况可靠性"极其敏感。这与 pass@k("至少一次成功")正相反。

长程套件把每个任务的关键步骤标为 critical 里程碑;longhorizon.score_passes 只在全部 N 遍都满足所有关键里程碑且不触发关键惩罚时记 pass^k=1,否则坍塌为 0,忠实于 τ-bench 语义。三个任务刻意构造成非对称:Fusion 在"自动售货机 12 周运营"上触发一致性漂移而失败,单模型在"退款政策严格执行"上越界而失败——

诚实的发现 于是长程 pass^3 打成 67% vs 67%(相当),里程碑覆盖 Fusion 略高(100% vs 94%)。 结论因此写成"可靠性相当,按成本而非可靠性决定"——这正复现了文章的告诫,而不是把 Fusion 吹成万能。

3.4 套件三:半导体 / DRAM 专项

以 2026 内存超级周期为背景,事实标准全部以一手资料接地:SK hynix 以约 50–62% 领跑 HBM;JEDEC 2025 年 4 月发布 HBM4 规范,接口翻倍到 2048 位、每堆栈约 2 TB/s,SK hynix HBM4 数据率约 10 GT/s、2026 年初放量;DDR5 6400 MT/s 主流;中国 CXMT 与美国出口管制构成供给/地缘风险。

3.5 反馈、评分与结论

每次评测产出结构化报告(render_markdown / render_json),包含:总分与相对 lift、按类别 / 按领域的明细、双方各触发哪些惩罚/错误标准、成本倍数 / 延迟倍数 / quality-per-dollar,以及一个结论性 verdict。

04

Skills 集成

核心立场:skill 是评测通过之后,才被拿来直接应用的那一部分。评测是"决定要不要用、何时用"的闸门;skill 是"决定怎么用"的产物。两者职责分明——评测产出 verdict,verdict 设定触发阈值,skill 在阈值满足时被调用。

4.1 评测如何为应用设闸

  • strong_win / win 的领域 → 把这类问题(架构决策、深度研究)列入"值得融合"清单。
  • budget_win → 高并发、成本敏感场景的默认面板选预算盘。
  • marginal / neutral / regression → 该类问题直接走单模型,Fusion 不介入。
  • 长程 → verdict 既然说"可靠性相当",就按成本决定,而不是默认上 Fusion。

4.2 作为工具被模型自己调用

Fusion 的设计哲学是"模型在问题值得时才自己去调它"。集成时把 open-fusion 注册成一个 server tool / 函数,让模型用一个 question 参数触发;SKILL.md 的 description 已调成在"fuse / 多模型共识 / 第二意见 / MoA"等措辞上触发。保持选择性——它慢约 2–3 倍且累加计费,是给研究/决策用的,不是闲聊。

# Claude Code / Codex:把本目录作为 skill,模型按需调用
open-fusion "为多租户 API 设计一个限流器" --preset quality --show-analysis

# 库用法:先评测,后应用
from open_fusion import fuse, load_preset
r = await fuse(question, load_preset("quality"))
print(r.text)          # 最终答案
print(r.telemetry)     # 补全数 / token / 关键路径ms / 状态

4.3 成本与延迟的预期管理

预设相对成本定位
quality≈ 3×+ 单前沿模型买质量天花板,不是省钱
budget≈ 0.4–0.5× 单前沿模型逼近前沿且更便宜
—延迟 ≈ max(面板)+裁判+合成≈ 2–3× 单次
